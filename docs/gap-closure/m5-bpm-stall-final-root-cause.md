# M5 BPM 停更问题最终结论

## 背景

在本轮联调中，最终已经验证：

- iOS 端 BPM 现在可以稳定跟随
- waveform 也可以持续更新
- `Manual` 模式下手动调整 HR 后，iOS 侧能继续同步变化

在问题真正闭环后，需要把“这次问题的本质到底是什么”说清楚，避免后续把两个不同层级的问题混为一谈：

1. **最终导致这次 BPM 停更现象的主根因**
2. **链路上同时存在、但不是这次最终主根因的工程缺口**

## 最终结论

### 结论一句话

这次 “iOS 端 BPM 不更新，但 waveform 仍持续更新” 的**最终主根因**，不是背压本身，而是**协议重组层 `FrameAssembler` 的重复帧判定过于粗糙，只按 `seq` 去重，误伤了心率数据帧**。

同时，`notify backpressure` 没有被正确处理，确实也是一个真实存在的链路风险，但它属于：

- **工程健壮性缺口**
- **高负载场景下的真实风险点**
- **不是这次最终导致 BPM 持续停更的直接主根因**

## 为什么最终结论不是“背压是主因”

第一阶段排查时，之所以会优先怀疑背压，是因为现象上非常像：

- 连接还在
- 外设本地 HR 还在变
- waveform 有流量
- iOS 端的 BPM 却停在旧值

这类现象在 BLE 业务里，确实常见于：

- notify 缓冲打满
- 发送端未做 pending queue
- `peripheralManagerIsReady(toUpdateSubscribers:)` 未实现

所以把背压当作优先怀疑方向是合理的，它不是误判，而是一个**正确但不完整**的中间结论。

但是在修完最小版 notify 背压处理之后，进一步联调发现：

- waveform 仍然稳定进入 iOS
- 但 heartRate 仍可能不更新

这说明问题不只是“发不出去”，而是：

- **有些帧已经到了 central**
- **但在协议重组 / 去重阶段被错误丢弃了**

也正因为这样，最终根因才收敛到了 `FrameAssembler`。

## 最终主根因：协议层重复检测误伤

### 旧实现的问题

`FrameAssembler` 原实现中，已完成 frame 的重复检测只基于：

- `seq`

也就是说，只要某个 `seq` 被视为“已经见过”，后面再次出现同样 `seq` 的单帧数据，就可能直接被当作 duplicate 丢掉。

这在只有单一路径、单一 payload 类型时问题不明显，但在当前项目里并不成立，因为：

- heartRate sample 会编码成 `.data(DeviceSample)`
- waveform block 也会编码成 `.data(WaveformBlock)`
- 它们都会走同一条协议重组器
- 它们的 frame seq 又可能在运行时重用或碰撞

于是就出现了一个关键风险：

- waveform 先占用了某个 `seq`
- 后续 heartRate sample 也用了同样的 `seq`
- `FrameAssembler` 只按 `seq` 去重
- heartRate frame 被误判为 duplicate
- App 端拿不到新的 `.data(DeviceSample)`
- `heartRateReceived(...)` 不再触发
- BPM 停在旧值

### 为什么 waveform 还能继续有

这是这次问题最容易让人困惑的地方。

因为从现象看：

- waveform 还在更新
- 但 BPM 不动

这正是协议层误伤的典型表现：

- 并不是整个链路都死了
- 而是**某一类 frame 被选择性误杀了**

当前实现中，waveform 和 heartRate 虽然都走 data frame，但它们的 payload 特征不同；如果去重逻辑只看 `seq`，就会出现“某一条流误伤另一条流”的问题。

所以，这不是“连接断了”，也不是“全部数据都没到”，而是：

- **协议重组器把合法的新 heartRate frame 当成了旧 frame**

## 这次真正修复了什么

### 修复 1：把重复检测从“只认 seq”改成“认 frame identity”

在 `FrameAssembler` 中，重复检测从：

- `seenSeqs`

改成了：

- `seenFrameIdentities`

当前使用的 identity 包括：

- `seq`
- `payloadCount`
- `payloadPrefix`

也就是说，只有在以下条件同时成立时，才会被视为真正重复帧：

1. `seq` 相同
2. payload 长度相同
3. payload 前缀签名相同

这样就避免了：

- 不同 frame 类型
- 不同 payload 内容
- 但刚好复用同一个 `seq`

时被误判为 duplicate。

### 修复 2：增加协议层回归测试

新增了覆盖以下场景的回归测试：

- 同一个 `seq`
- 先收到 waveform
- 再收到 heartRate data
- 两者都必须被成功解码

这类测试的价值非常高，因为它直接把这次事故根因固化成了协议契约，能避免后续再被类似“简化去重逻辑”的改动破坏。

## 背压问题在这次事故里的定位

### 背压问题是什么

`notify backpressure` 指的是：

- `CBPeripheralManager.updateValue(...)` 返回 `false`
- 当前发送窗口已满
- 发送端需要显式缓存并等待 `peripheralManagerIsReady(toUpdateSubscribers:)`

之前 Simulator 侧确实没有做好这部分，因此已经补了：

- pending queue
- `isReady` 后继续 drain
- 心率优先级高于 waveform

### 背压为什么不是这次最终主根因

因为在补完最小背压处理后，问题并没有完全解释完。

最终能让 BPM 真正恢复稳定跟随的关键修复，是协议层去重逻辑的调整，而不是仅仅加队列。

所以这次问题要分层理解：

- **背压缺失**：链路健壮性问题，是真实风险，需要修
- **协议层按裸 `seq` 去重**：这次 BPM 停更的最终主根因

### 那背压还重要吗

非常重要。

在真实业务里，如果没有背压处理，即使协议层去重没问题，也仍然会出现：

- 高流量场景下关键数据晚到
- 心率、状态事件、告警被波形挤压
- 前后台切换时链路抖动放大

所以这次最终结论不是“背压不重要”，而是：

- **背压问题是真实存在的工程缺口**
- **但这次具体事故最终卡住 BPM 的最后一击，是协议去重误伤**

## 问题本质的最终表述

这次问题如果要用一句更准确的话来描述，应该是：

> **表面现象是 BPM 停更，表层怀疑是 notify 背压，但最终主根因是协议重组层对 frame duplicate 的判定过粗，只按 `seq` 去重，导致 heartRate frame 被 waveform 流误伤。**

## 对实际业务的启示

这次问题非常典型，说明 BLE 实时业务不能只盯某一层。

### 启示 1：连接状态不代表数据语义正确

业务上经常容易误以为：

- 连接还在
- waveform 还有数据
- 所以心率链也应该是通的

但真实情况是：

- 连接层正常
- 一部分数据流正常
- 另一部分数据流在协议层被误伤

所以必须把“连接健康”和“数据语义健康”分开看。

### 启示 2：去重逻辑不能过于简化

协议实现里最容易出事故的逻辑之一，就是：

- 为了防 duplicate/retransmit
- 写了一个看似简单的判定
- 但 identity 定义过窄

这次就是典型案例。

在多 payload、多流复用同一通道时，去重 identity 至少要能区分：

- frame 语义
- payload 特征
- 必要时区分不同 data kind

### 启示 3：背压治理和协议正确性必须同时做

这次问题最终闭环后，最重要的经验不是“以后只查协议层”，而是：

- **发送端要做背压治理**
- **接收端要做正确的 frame identity 判定**

两者缺一不可。

## 当前结论

最终可以明确回答这个问题：

### 1. 这次问题本质是什么？

**最终主根因是协议部分的错误处理，更准确地说，是 `FrameAssembler` 的 duplicate 判定过粗，错误地把合法的 heartRate frame 当成了重复帧。**

### 2. 那是不是没有做背压处理？

**也是。**

但它在这次事故中的定位是：

- 真实存在的链路治理缺口
- 高负载下会导致丢包或延迟
- 不是这次最终让 BPM 停更的直接主根因

### 3. 最终应如何定性？

最准确的定性是：

- **主根因：协议层 duplicate 判定错误**
- **并存缺口：发送端未完整处理 notify 背压**

这两类问题都需要修，而当前已经完成了对应修复。
