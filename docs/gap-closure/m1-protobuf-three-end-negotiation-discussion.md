# M1 · 用 Protobuf 最大化降低三端协议协商成本的讨论

## 1. 文档目的

本文件只做架构讨论，不做任何代码落地。

目标是回答一个更深的问题：

- 如果项目希望 **极大降低 iOS / Android / 固件 三端在硬件协议上的协商成本**
- 那么是否可以认为 **Protobuf 应该在项目中被大规模使用**
- 如果可以，应该用到什么程度
- 如果不应该全量使用，边界在哪里

这不是“要不要接入 Protobuf”的问题，而是：

- **在多端协作成本最小化这个目标下，Protobuf 最优使用范围是什么**

---

## 2. 问题本质

三端硬件协议协商成本高，通常不是因为“序列化格式不够先进”，而是因为下面四件事同时存在：

1. **协议语义分散**
   - iOS 一套理解
   - Android 一套理解
   - 固件一套理解
   - 文档再单独一套表述

2. **字段演进靠人工同步**
   - 新增字段时，需要分别改 Swift / Kotlin / C/C++
   - 字段号、可选性、默认值、废弃字段都靠人工对齐

3. **版本兼容规则不统一**
   - 一端允许跳过未知字段
   - 另一端要求强校验
   - 第三端直接忽略或报错

4. **协议讨论没有单一事实来源**
   - 文档是讨论
   - 代码是现实
   - 生成产物不是契约

因此，真正想降低三端协商成本，目标不应该表述成：

- “多用一点 Protobuf”

而应该表述成：

- **让跨端协议契约只有一个主来源，并让大部分结构化语义由 schema 自动表达**

如果按照这个目标来判断，Protobuf 的价值不是“二进制更高级”，而是：

- **它能把跨端协议讨论从‘口头 + 文档 + 手写解析器’收敛成‘schema + 生成代码 + 版本规则’**

---

## 3. 什么叫“在项目中极大使用 Protobuf”

这里先给出一个清晰定义。

“极大使用 Protobuf” 不等于：

- 把所有 BLE 负载都改成 Protobuf
- 把 GATT / 分帧 / ACK / CRC 都改写成 Protobuf

“极大使用 Protobuf” 更准确的含义应该是：

- **让绝大多数跨端需要长期协商的结构化语义，都由 `.proto` 承担**

也就是：

- 控制面消息大量由 Protobuf 表达
- 状态面消息大量由 Protobuf 表达
- 低到中等吞吐的数据容器尽量由 Protobuf 表达
- 只有极少数对吞吐和位级布局极端敏感的路径，继续保留自定义紧凑编码

换句话说：

- **“极大使用”指的是语义覆盖率高**
- **不是字节流百分之百都必须由 Protobuf 编码**

---

## 4. 可以把 Protobuf 用到什么程度

为了避免讨论抽象化，可以把使用程度分成 5 个等级。

### 等级 0：只讨论，不落地

特点：

- 文档里认可 Protobuf
- 代码不接
- schema 不存在

结果：

- 三端协商成本几乎不变

这类状态不具备实际意义。

### 等级 1：只在少量低风险响应中使用

典型范围：

- `HELLO_ACK`
- `INFO`
- `DeviceInfo`

结果：

- 可以验证工具链
- 可以验证双端灰度
- 但无法显著降低三端整体协议协商成本

原因：

- 被 Protobuf 接管的语义面太窄

这更像“工程可行性验证”。

### 等级 2：接管控制面与状态面

典型范围：

- `HELLO`
- `HELLO_ACK`
- `INFO`
- `ERROR`
- `CONFIG`
- 设备状态
- 设备事件
- 能力声明
- 参数协商

结果：

- 三端协议讨论成本会显著下降
- 因为大部分“最容易争议、最容易扩展”的消息都已经由 schema 表达

这是第一个真正有战略价值的等级。

### 等级 3：接管控制面、状态面，并接管中低吞吐数据容器

典型范围：

- 控制面全部
- 状态面全部
- 样本批量容器
- 诊断导出消息
- 算法结果消息
- 睡眠阶段或指标上传消息

结果：

- 三端协议协商成本进一步下降
- 因为除了极少数高吞吐路径，几乎所有跨端语义都可以在 `.proto` 层完成

这时项目已经进入“Protobuf 是跨端协议主语言”的状态。

### 等级 4：试图接管全部数据，包括高吞吐原始块

典型范围：

- 波形块
- OTA 数据块
- 高频原始采样
- 可能还包括低层 ack 风格消息

结果：

- 协商表面上统一了
- 但性能、MTU、编码密度、MCU 资源占用、调试复杂度会急剧上升

这是一个看起来统一、实际上风险很高的等级。

### 等级 5：连传输层思维都试图交给 Protobuf

表现：

- 试图让 Protobuf 直接主导 GATT / 分帧 / ACK 设计

结论：

- 这是错误方向
- 它会破坏当前项目最核心的分层边界

---

## 5. 对本项目而言，最合理的目标等级

本项目当前最合理的长期目标不是等级 4，也不是等级 5。

最合理的目标是：

- **等级 3 的上半段**

即：

- 让 Protobuf 成为 **控制面 + 状态面 + 结构化结果面** 的主表达方式
- 同时保留对高吞吐、位级优化路径的自定义编码

更具体地说：

### 应该尽量交给 Protobuf 的部分

1. 握手元数据
2. 能力协商
3. 配置下发
4. 设备信息
5. 错误模型
6. 结构化事件
7. 结构化状态
8. 诊断与导出消息
9. 算法结果
10. 睡眠/HRV/推理这类结构化业务数据容器

### 不应强行交给 Protobuf 的部分

1. GATT 服务与 Characteristic 边界
2. L2 分帧 / CRC / ACK / seq
3. 波形高吞吐原始块
4. OTA 数据块
5. 极小、高频、对字节成本非常敏感的实时样本主链

这意味着：

- **如果目标是极大降低三端协商成本，Protobuf 应该成为项目中绝大多数结构化协议语义的主承载体**
- **但它不应该成为所有字节的主承载体**

---

## 6. 为什么不是“全量 Protobuf”最优

很多团队会自然滑向一个简单判断：

- “既然 Protobuf 能减少协商，那就应该把全部协议都变成 Protobuf”

这个判断在本项目里不成立。

原因有 5 个。

### 6.1 协商成本和传输成本不是同一个问题

Protobuf 擅长解决：

- 字段协商
- 模型一致性
- 版本兼容
- 多端代码生成

但它不擅长解决：

- BLE MTU 限制
- 高频流的最小字节成本
- 波形块的极限吞吐
- OTA 数据传输效率

如果把协商问题和传输问题混成一个问题，方案会偏。

### 6.2 固件端不是无限资源环境

在 iOS / Android 上，Protobuf 的运行时成本往往不是核心问题。

但固件端不同：

- code size
- stack / heap
- parse cost
- upgrade complexity

都会直接进入约束。

### 6.3 高吞吐路径更适合位级优化

波形、OTA、原始采样块的价值不在“字段多”，而在：

- 量大
- 频高
- 对编码密度敏感

这些路径继续保留紧凑自定义格式，通常比强行统一成 Protobuf 更合理。

### 6.4 全量切换会放大迁移面

当前仓库已经有稳定的 TLV 与自定义编码实现。

如果全量切换：

- 风险会同时扩散到 M1 / M3 / M5 / M6
- 每条现有联调链都要重新验证
- 任何问题都更难定位到底是 schema、编码、分帧还是业务逻辑错误

### 6.5 “全量统一”不等于“总体更优”

架构最优不是追求单一技术绝对纯度，而是：

- **让每一层用最适合自己的表达方式**

对本项目来说：

- 结构化语义最适合 Protobuf
- 吞吐与位级优化最适合紧凑自定义编码

---

## 7. 如果真正想把三端协商成本压到最低，应该怎么做

如果把“最小化三端协商成本”作为第一目标，那么关键不是“是否全量 Protobuf”，而是建立下面这套机制。

### 7.1 把 `.proto` 变成跨端协议单一事实来源

应该做到：

- Swift / Kotlin / Firmware bindings 都从 `.proto` 生成
- `.proto` 先于实现变更
- 所有结构化协议讨论先落在 schema

一旦做到这一点：

- 三端在字段层面的讨论会明显下降

### 7.2 把 TLV 从“长期主契约”降级为“过渡与特例承载”

对本项目而言，更合理的长期状态不是：

- TLV 和 Protobuf 永久平权

而是：

- Protobuf 成为结构化协议主契约
- TLV 或自定义紧凑编码只保留在：
  - 迁移过渡区
  - 高吞吐特例区

### 7.3 把“消息模型协商”前移到 schema 评审，而不是联调现场

当三端联调时才讨论：

- 字段有没有
- 默认值是什么
- 未知字段怎么办

说明 schema 机制没有真正建立。

目标状态应该是：

- 联调只验证实现
- 不再现场重新定义契约

### 7.4 用 capability 解决“能不能用”，用 schema 解决“如何表达”

能力位应该只负责：

- 对端是否支持 protobuf 分支
- 是否支持某类功能

`.proto` 应该负责：

- 字段定义
- 可选性
- oneof
- reserved
- 演进规则

这两个层级不能混用。

### 7.5 把 breaking change 控制前置

真正降低三端沟通成本，关键不是“版本号递增”，而是：

- 防止无意 breaking changes 混入

长期上应建立：

1. schema review
2. breaking check
3. 生成产物回归
4. golden byte tests

这样团队讨论会从“这次怎么又坏了”变成“这次 schema 是否允许这样演进”。

---

## 8. 在项目里，Protobuf 最终可以用成什么程度

这里给出一个更直接的结论。

### 可以用到的程度

如果项目的核心目标是减少三端协商成本，那么 Protobuf 最终可以用到：

- **几乎覆盖所有结构化控制面与状态面协议**
- **覆盖绝大多数中低吞吐、语义复杂的数据容器**
- **成为三端协议讨论和版本演进的主要语言**

在这种状态下，项目团队看到协议时，第一反应应该是：

- 去看 `.proto`

而不是：

- 去翻 TLV tag 表
- 去翻三份平台实现
- 去翻聊天记录

### 不应超过的程度

它不应该进一步扩张到：

- 取代所有高吞吐紧凑块
- 取代 L2 传输设计
- 取代 BLE 层约束思维

换句话说，Protobuf 在本项目里最适合成为：

- **协议语义主语言**

而不是：

- **所有字节流的唯一承载体**

---

## 9. 一个更准确的长期架构表述

如果以后要对外或对团队内部描述目标状态，最准确的表达不是：

- “项目全面转向 Protobuf”

而是：

- **项目采用“自定义 BLE 传输层 + Protobuf 主导结构化协议语义 + 紧凑编码承载高吞吐原始数据”的混合架构**

这句话有 3 个重点：

1. 传输层仍是自定义 BLE 协议
2. 结构化语义主导权交给 Protobuf
3. 高吞吐原始数据保留最优编码策略

这才是最接近“既降低三端协商成本，又不牺牲性能与稳定性”的状态。

---

## 10. 对当前项目的实际建议

如果只从讨论层给结论，不谈立即实施，那么建议是：

### 建议一

把 Protobuf 的长期定位从：

- “可选增强”

升级为：

- **结构化跨端协议的主契约候选**

### 建议二

不要把目标设成：

- “TLV 全量迁移到 Protobuf”

而要设成：

- **除高吞吐特例外，绝大多数跨端结构化语义逐步迁移到 schema-first 模式**

### 建议三

把未来的协议讨论分成两层：

1. **结构化语义层**
   - 默认优先 Protobuf
2. **吞吐与传输优化层**
   - 默认优先紧凑自定义编码

### 建议四

如果未来真的希望三端协商成本进一步大幅下降，最终追求的不是：

- “项目里用了很多 Protobuf”

而是：

- **项目里大部分跨端协议讨论都不再需要人工重新定义字段**

只要这一点成立，才说明 Protobuf 真正发挥了价值。

---

## 11. 最终结论

对于本项目，“为了极大减少三端协商成本，是否应该在项目中极大使用 Protobuf”的答案是：

- **是，但不是全量替换意义上的极大使用**

更准确的结论是：

- **应该让 Protobuf 成为项目中绝大多数结构化协议语义的主契约表达**
- **不应该让它取代高吞吐原始块和传输层优化路径**

因此，最优认知不是：

- “Protobuf 应不应该全项目化”

而是：

- **Protobuf 应该主导语义层，紧凑编码继续主导吞吐层**

如果这个边界保持清晰，项目可以同时获得：

1. 三端协商成本显著下降
2. 协议演进成本下降
3. 代码生成一致性提升
4. 高吞吐路径性能不被破坏
5. 架构边界仍然稳定
