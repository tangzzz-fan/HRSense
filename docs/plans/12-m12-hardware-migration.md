# M12 · 真机迁移 — 实施计划

## 摘要

从模拟器切换到真实硬件。核心约束：**App 层对变化基本无感知** —— 差异通过协议层已有的能力协商机制吸收。模拟器必须保留为 CI/回归工具。

---

## 前置条件

| 资产 | 来源 | 说明 |
|---|---|---|
| 共享协议包 `HRSenseProtocol`（单测 ≥80% 覆盖率） | M1 | L2-L4 编解码 |
| macOS 模拟器（场景引擎、故障注入、headless） | M2/M5/M6/M11 | 全特性外设 |
| iOS App（Clean+Redux、BLE、波形、OTA） | M3/M4/M5/M6 | 全特性中心 |
| 场景脚本库（JSON） | M11 | 可复用 E2E 资产 |
| 可观测性（日志/指标/MetricKit） | M7 | 诊断基础设施 |
| **真实硬件板** | 外部 | 物理前提 |

---

## 阶段 1：发现与审计（零代码改动）

### 1.1 硬件协议文档审查

获取真实硬件 BLE 协议规范。如缺失，用 LightBlue/nRF Connect 反向工程。

### 1.2 分层协议差异矩阵

按 L1（GATT 传输）、L2（分帧/可靠传输）、L3（会话/命令）、L4（应用数据）逐层对照 `docs/03-ble-gatt-protocol.md`。

### 1.3 差异分类框架

| 分类 | 含义 | 处理方式 |
|---|---|---|
| **A 类** | 已由现有能力位吸收 | 无需改动 |
| **B 类** | 通过协议版本或 TLV 扩展吸收 | `HRSenseProtocol` 新增编解码路径 |
| **C 类** | 需 `HRSenseProtocol` 适配层 | 在协议包内编写翻译逻辑 |
| **D 类** | 需 App 感知 | **目标为零个** |

---

## 阶段 2：协议对齐与适配

### 2.1 能力协商验证

- 审计 App 层所有 `capabilities` 消费点，确认使用协商交集
- 扩展 `HRSenseProtocolTests` 能力矩阵（参数化测试）
- 创建能力协商集成测试场景

### 2.2 协议差异调和

对每处 B/C 类差异：先更新 `03` 契约 → 更新 `HRSenseProtocol` → 刷新 golden bytes 测试。

### 2.3 连接与 MTU/时序差异

- 验证 `BLECentralDataSource` 的分片逻辑针对硬件 MTU
- 使重连/超时参数可配置

---

## 阶段 3：App 层加固（移除模拟器硬编码假设）

在 `HRSenseFeature`、`HRSenseData`、`HRSenseCore` 中搜索并消除：
- 硬编码设备型号/固件字符串
- 假设特定能力位常驻
- 硬编码 MTU/采样率
- 硬编码扫描超时

---

## 阶段 4：E2E 关键路径在真实硬件上集成

- 连接与握手表征（M3 级别）
- 波形流表征（M5 级别）
- OTA/DFU 表征（M6 级别）

---

## 阶段 5：模拟器保留与回归测试迁移

### 5.1 真实硬件数据录制管道

- 连接硬件 → 订阅通知 → 录制成 CSV + JSON manifest
- 波形块录制成二进制格式

### 5.2 重放回归测试

- 加载录制数据集 → 以忠实时序重放给 App
- Golden bytes 合约测试覆盖真实硬件字节

### 5.3 CI 集成

- 模拟器专用 E2E（每次推送）
- 真实硬件回归（手动/定时触发）
- 破坏 golden bytes 测试的变更不得合并

### 5.4 场景脚本兼容性标记

```json
{
  "scenario": "ota_normal_upgrade",
  "compatibleBackends": ["simulator", "hardware-v2"],
  "requiresCapabilities": ["HEART_RATE", "OTA_DFU"]
}
```

### 5.5 模拟器退化防护

CI 冒烟测试验证模拟器在每次推送后仍可正常广播/连接/握手/推送数据。

```
  → 阶段 2（协议适配） — 12-40 小时

```
