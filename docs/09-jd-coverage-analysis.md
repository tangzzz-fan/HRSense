## 差距详解与补充方案

> 每项给出：**为何是难点 → 在本项目如何补充（模拟器可自证）→ 落到哪个文档/spec**。

### 3.1 实时波形高吞吐流（最高优先）
- **难点**：波形(如 ECG/PPG，250–500Hz)数据量远大于 HR bpm，考验 MTU 最大化、连接参数、notify/WriteWithoutResponse 吞吐、分片/重组、**背压与丢帧策略**、UI 不卡。
- **补充**：
  - 协议加**波形数据通道**：新增 `DataKind=波形`，批量样本 + `sampleSeq` 连续性；最大化 MTU、按 `min(MTU)-头部` 打包、连续 notify。
  - 吞吐优化清单：请求更大 MTU、缩短连接间隔(说明 iOS 限制)、批量打包降帧头开销、后台队列解码、环形缓冲 + 降采样上屏。
  - **模拟器**加"波形发生器"(合成 ECG/PPG)与**吞吐/丢包统计**面板；App 侧统计有效码率、端到端时延、丢样率。
- **落点**：扩展 `03`（波形 DataKind + 吞吐章节）；`05` 增波形发生器；新增 `spec 0003 · 波形高吞吐传输`。

### 3.2 睡眠结构分析（健康数据 + 加分项）
- **难点**：睡眠分期需长时段数据 + 特征(HR/HRV/体动趋势) + 分类模型。
- **补充**：
  - 复用现有管线：C++ 特征(spec 0001) → CoreML(spec 0002) 增一个**睡眠分期任务**(Wake/Light/Deep/REM 简化版)。
  - 模拟器 `replay` 一夜合成数据；App 出睡眠结构图(睡眠阶段带状图)。
- **落点**：`06` 增睡眠任务；`spec 0002` 扩展任务集；模拟器数据集加"整夜"场景。

### 3.3 本地持久化存储
- **难点**：长时序健康数据的存储、查询、聚合(日/周/月)、迁移。
- **补充**：
  - 选型：**SwiftData / Core Data**(结构化+查询) 或 **SQLite(GRDB)**；波形等大块可存文件 + 元数据入库。
  - 分层：`HRSenseData` 增 `PersistenceStore`(Repository 之下)；写入走后台，读走查询接口。
  - 保留策略/降采样归档(原始波形短期、聚合指标长期)。
- **落点**：新增 `spec 0004 · 本地存储与数据保留`；`04`/`08` 增持久化模块。

### 3.4 高性能可视化
- **难点**：实时波形/长趋势渲染，SwiftUI `Charts` 对高频波形可能吃力。
- **补充**：
  - 分层：趋势/HRV 用 **Swift Charts**；**实时波形**用**自绘**(`Canvas`/`CAShapeLayer`/Metal) + 环形缓冲 + 固定窗口滚动。
  - 明确刷新预算(≤这一档帧率)、降采样(min/max 抽样保形)、离屏/复用。
- **落点**：`04` 增"可视化策略"小节；或并入 `spec 0003`。

### 3.5 崩溃分析 / 日志 / 监控
- **难点**：可诊断的分层日志 + 崩溃/性能采集 + 指标监控，是"区分 App/BLE/固件"的基础设施。
- **补充**：
  - **结构化分层日志**：用 `OSLog`/swift-log，分类(BLE 字节流 / 帧 / 命令 / 状态 / OTA)，可导出便于两端对齐。
  - **崩溃&性能**：`MetricKit`(系统级、无三方) + 可选 Crashlytics/Sentry。
  - **监控指标**：连接成功率、重连次数、命令超时率、吞吐/丢样率、OTA 成功率——本地埋点 + 可视化面板(调试用)。
- **落点**：新增 `docs/10-observability.md`（日志/崩溃/监控）。

### 3.6 数据协议：自定义栈 vs Protobuf
- **现状**：项目按需求用**自定义二进制帧**(TLV)。JD 点名 **Protobuf**。
- **补充/取舍说明**：
  - 二者不冲突：可让 **L4 应用数据层的 payload 采用 Protobuf 编码**，L2 分帧/可靠传输保持不变——即"自定义传输 + Protobuf 负载"。
  - 在 `03` 增一节说明：为何自定义栈(私有命令/分片/版本协商)，以及**如何以 Protobuf 承载 L4**(定义 .proto、生成 Swift、与 FW 共享 schema)。这样既满足"自定义 GATT 栈"要求，又能展示 Protobuf 经验。
- **落点**：`03` 增"L4 可选 Protobuf 承载"小节；可加 `proto/` 目录规划到 `08`。

### 3.7 后台 BLE / 后台任务
- **难点**：iOS 后台运行 BLE(central background mode)、**状态恢复(State Preservation & Restoration)**、后台断线重连、系统限制。
- **补充**：
  - `04`/`08` 明确：`UIBackgroundModes: bluetooth-central`、`CBCentralManager` 的 restoreIdentifier + `willRestoreState`、后台重连策略与限制。
  - 模拟器可配合"长时间静默 + 突发数据"验证后台唤醒路径(需真机后台实测)。
- **落点**：`04` 增"后台与状态恢复"小节；App 外壳 entitlements/Info.plist(`08`)。

### 3.8 读 Python/MATLAB 原型 → 移植
- **补充**：在 `06`/`spec 0001` 增"算法移植流程"：算法工程师给 Python/MATLAB 原型 → 明确 I/O 与数值口径 → 移植为 C++(计算) 或转 CoreML(模型) → 用原型输出做**黄金值对拍**。
- **落点**：`spec 0001`/`06` 增小节。

## 5. 结论

- **已是强项**：OTA/DFU、GATT 自定义栈/裸字节流、断线重连、C/C++ 桥接、Core ML、"区分 App/BLE/固件"的可诊断性(模拟器+故障注入)。
- **主要差距**：实时波形高吞吐、可观测性(崩溃/日志/监控)、后台 BLE、本地存储、高性能可视化；次要：睡眠结构、Protobuf 承载、原型移植。
- **补齐后**：项目可较完整地对应 JD 的每一条职责与必备/加分能力，且全部可在**无硬件、macOS 模拟器**条件下自证。

> 说明：本文只做分析与规划。是否落地各补充项(新增 spec 0003/0004、docs/10、以及对 03/04/06/08 的小节扩展)，可按上表优先级逐步进行。
